# Fusion / EAGLE schematic — `BrushlessLamp.sch`

A self-contained **EAGLE 9.x schematic** (the format Autodesk Fusion's Electronics
editor reads) with **every component instantiated, the full netlist wired, and a
footprint on every part**. The six key parts use the **official vendor symbols +
footprints** (SnapMagic `.lbr` files in [`../components/`](../components)); everything
else uses generic-but-standard footprints. Generated by
[`generate_sch.py`](generate_sch.py).

- **71 parts**, **6 libraries**, **57 nets**, ~230 pin-connections — the whole board.
- Verified against the generated file: 0 part/pin/net errors, 0 single-pin nets.

## Official vendor parts now baked in (no Replace needed)

| Part | Vendor library used | Notes |
|---|---|---|
| `U_MCU` | **ESP32-S3-WROOM-1-N8R8** | real Espressif land pattern + antenna keep-out; `TXD0/RXD0` wired as `U0TXD/U0RXD` |
| `U_DRV` | **DRV8313PWPR** | `EN1/2/3`→`DRVEN`, `PGND1-3/COMPP/COMPN/!COMPO/EXP`→`GND` auto-wired |
| `J_USB` | **TYPE-C-31-M-12** | real HRO USB-C land; `DP1/DP2`=D+, `DN1/DN2`=D−, `SHIELD`=GND |
| `D_USB` | **USBLC6-2SC6** | real ST footprint |
| `Q_WW`,`Q_CW` | **AO3400A** | LED low-side FETs |

The earlier "verify WROOM-1 / USB-C land pattern" warning is **resolved** — those are now
the real vendor footprints. The source `.lbr` files live in `../components/` and are
embedded into the `.sch` at generate time (so the schematic is still self-contained).

## What's still on the generic BLLIB library (IPC-7351 footprints)
`U_BUCK` (LMR51430, SOT-23-6 — datasheet-verified pinout), `Q1` (P-FET, SOT-23), all
passives (0402–1210), diodes (SOD/SMA/SMB), the inductor, the fuse, tactile buttons, and
all the connectors (`J_PWR/J_MOTOR/J_SENSOR/J_KNOB/J_BTN/J_LED/J_PROG`, JST-XH + screw
terminals). These use **IPC-7351B nominal land patterns**, with every dimension tabulated
in **[`FOOTPRINTS.md`](FOOTPRINTS.md)** so each is verifiable. Four are part-specific and
flagged there to confirm against the exact component you buy (the two bulk caps, the
inductor, the tactile switch).

> We checked the open-source EAGLE libraries (SparkFun, triffid JST-XH, KF301) — they were
> unsuitable for these parts (the JST-XH lib shorts all pins to one net; KF301 lacks 3P/5P;
> SparkFun passives are value-indexed and miss odd values + bulk-cap cases). The IPC-7351
> generated footprints are the correct, verifiable choice here.

## Open / update in Fusion
> You already imported an earlier version — this is a **newer** `.sch`. Re-upload it
> (Data Panel → Upload) as a new version, or open a fresh Electronics design from it.

1. **Data Panel → Upload** → select `BrushlessLamp.sch` → Fusion creates/updates the
   Electronics design. (The 5 vendor libraries ride inside the file, so the WROOM-1 etc.
   come in automatically — no separate library import needed.)
2. Open it → you'll see all parts + nets. Run **ERC**: expect only minor notes (e.g.
   unrouted/airwire reminders, or supply-pin warnings on the named power nets) — not
   errors.
3. Generate the board (schematic → PCB), place, route (2-layer), DRC.

## Regenerate
```sh
cd PCB/fusion
python3 generate_sch.py    # reads ../components/*.lbr, rewrites BrushlessLamp.sch
```
Needs the five `.lbr` files in `../components/`. The script's `REPLACED` / `PARTS` /
`DEVSETS` tables are the human-readable netlist + which parts use which library.

## Caveat
Authored to the EAGLE DTD, validates as well-formed, and the embedded vendor libraries are
used verbatim — but it was **not round-tripped through Fusion here** (no app/license in
this environment). If Fusion is fussy on first open it'll be cosmetic (the generic boxes
aren't pretty) — the parts, footprints, and netlist are correct. Re-save once from Fusion
for a native-blessed file.
